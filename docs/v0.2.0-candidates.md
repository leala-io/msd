# MSD v0.2.0 Candidates — gaps surfaced by the v0.1.0 build

This log records real-world structures encountered while validating the mybuxi Emmental example against the
v0.1.0 schema that the **§3 data model cannot express**. Per the build blueprint freeze (§8), these are **not**
built in v0.1.0. They are recorded here as candidates for a future Concept Paper revision and v0.2.0 schema.

The v0.1.0 schema is intentionally open (`additionalProperties` is not `false`), so the example carries these
structures as pass-through extensions and still validates. Nothing below is silently added to the schema.

---

## A. Open-schema extensions present in the mybuxi example (Decision 1)

These nine objects are out of §3 scope. They validate today only as pass-through extensions and need a modelled
representation in v0.2.0:

| # | Structure | Where | Notes |
|---|---|---|---|
| 1 | `swiss_profile` | provider, service | MSD-CH profile (Concept Paper Appendix D): SBOID, GO-Nummer, SLNID, konzession_typ, service_category. |
| 2 | fare `products` | fare_structure | Subscription/pass products: day_pass, multi_ride, monthly_pass, annual_pass; transferable vs personal. Not in §3.5 (single-trip pricing only). |
| 3 | fare `eligibility` | fare_structure | Conditional fare scoping (membership, municipality residence, partner trip; `default: true`). Needs a fare_group / fare_eligibility concept beyond simple discounts. |
| 4 | `partners` | root | Partner network with location-conditional reduced fares (e.g. CHF 5 to/from a partner stop). |
| 5 | `safety` | service | Driver identification, vehicle tracking, reporting mechanism, etc. |
| 6 | `sales_points` | fare_structure | Ticket distribution channels (vehicle, physical, email, webshop) and which products each sells. |
| 7 | `credential_acceptance` | fare_structure | Explicit **non-acceptance** of credentials (`ch:halbtax: false`, `ch:ga: false`). §3.5 models credential-based discounts but not negative declarations. |
| 8 | booking `multi_stop` | booking_rules | Multi-stop trip support flag. |
| 9 | booking `group_booking` | booking_rules | Group booking: max group size, child-accompaniment rules. |

Related fare gaps noted inside the example (subsets of the above): **group pricing** (flat price per vehicle up to
N passengers — carried in v0.1.0 as `discount.type: "flat"`) and **municipality-based zones** (administrative
boundaries rather than GeoJSON polygons).

## B. Richer shapes conformed away in the example (Decision 4)

For these two fields the build kept a single canonical §3 shape and conformed the example. The richer mybuxi shapes
are recorded here as Concept-Paper candidates:

1. **Object-form `booking_channels`.** §3.6 / v0.1.0 model `booking_channels` as an array of code-list **strings**
   (`["app","phone","email"]`). mybuxi originally carried objects with extra detail:
   ```json
   [
     { "type": "app",   "url": "https://mybuxi.ch/fahrgaeste/#App" },
     { "type": "phone", "description": "For group bookings and special destinations" },
     { "type": "email", "address": "emmental@mybuxi.ch", "description": "..." }
   ]
   ```
   v0.2.0 candidate: allow per-channel url/address/description.

2. **Rule-based (date-less) operating-hours exceptions.** §3.3 / v0.1.0 require `operating_hours.exceptions[]` to be
   **date-keyed** (`{ "date": "2026-12-25", "closed": true }`). mybuxi carried a recurring holiday rule with no date:
   ```json
   { "rule": "Allgemeine Sonn- & Feiertage: Sonntagsbetrieb" }
   ```
   This item was removed from the example (it was the only exception). v0.2.0 candidate: a `holiday_rule` / recurring
   exception concept in `operating_hours`.

## C. Service-area `stops` — added in v0.1.0 (Decision 2), noted for completeness

`service_area.type: "stops"` with a required `stops` array was **added** in v0.1.0 (permitted expansion, blueprint §8;
backed by Concept Paper Appendix D.1), correcting the blueprint's zones-only scope. The mybuxi example's
representative-sample field `stops_sample` (5 of 335 stops) was renamed to `stops` to conform; the full list remains
referenced via `stops_count` and `stops_url`. A future revision may distinguish a complete stop list from a
representative sample.

## D. Gaps surfaced by the Kampala minibus-taxi (matatu) SACCO example (Track E)

Surfaced while validating [`examples/ug/kampala-taxi-sacco.msd.json`](../examples/ug/kampala-taxi-sacco.msd.json)
(hypothetical ~30-vehicle single-corridor minibus-taxi SACCO) against the frozen v0.1.0 schema. The example **validates
PASS**; the items below are real-world structures the §3 model cannot express and are carried as pass-through extensions
(or omitted). Full feature → field → expressibility analysis: [`coverage-gap-matrix.md`](coverage-gap-matrix.md).
Sources: ITF Roundtable Report 199, GIZ paratransit-reform guidance, Oviedo/Bogotá reform-pattern studies (source-level;
citations to be tightened). Per the freeze (blueprint §9) these are **not** built in v0.1.0.

| # | Structure | Where | Notes |
|---|---|---|---|
| 1 | `operational_model` — **fill_and_go** | service | Depart-when-full + hail-and-ride boarding + no advance booking. The `service_type` registry has only `on_demand`; `booking_rules` assume advance booking. `service_type` set to `on_demand` as a forced placeholder. Relates to issues #4/#5/#6/#14. |
| 2 | `organisation_type` — **SACCO / cooperative** | provider | Owner-operator cooperative / route association; no `organisation_type` field in `provider`/`legal_entity`. |
| 3 | `dynamic_pricing` — varying / negotiated fares | fare_structure | Fares vary by time of day, weather, demand, and negotiation. `fare_rule.price` is a single fixed number. |
| 4 | hail-and-ride / board-anywhere along corridor | service_area | The `stops` model assumes discrete boarding points; continuous on-line boarding is not representable. Route **order** of stages is also not modelled. |
| 5 | informal **settlement** (driver daily target/fee, SACCO dues) | settlement | §3.7 settlement targets MaaS/platform settlement; an informal cash / daily-target revenue model has no fitting representation. |
| 6 | **diesel** propulsion (minor) | vehicle | The `propulsion` registry contains only `electric`; the optional field is omitted (no new code-list value). A future registry may add propulsion values. |
| 7 | **minibus-taxi (matatu)** vehicle class (minor) | service.mode | Modelled as `bus`; the minibus-taxi distinction is not separately expressible. |

**Note on `safety`:** the Kampala example also carries a `service.safety` object. This is **not** a new gap — it is the
same structure already recorded as **A.5** above; the Kampala data reinforces it (safety is the dimension reform
programmes target, per ITF-199 / GIZ).
